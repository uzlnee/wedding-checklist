# 웨딩 체크리스트 항목·시기 재구성 설계

작성일: 2026-06-18

## 목표

첨부된 종이 체크리스트(상견례 → D-1까지 8단계)를 기존 앱에 **병합**한다.
범용 템플릿으로 쓸 수 있도록, 상견례부터 결혼식 전날까지 모든 단계를 고르게 담는다.

기존 앱의 강점(항목별 업체명·예산·상태 관리, 카테고리 분류, 일정/예산 탭)은
그대로 유지하고, 사진에만 있는 누락 항목과 막판 시기(2주/1주/D-1)를 보강한다.

## 접근 방식: 병합 (Merge)

세 가지 다른 접근(병합 / 전면 교체 / 두 모드 제공) 중 **병합**을 택했다.
기존 데이터 모델(카테고리 + 업체 + 예산 + 상태)을 보존하면서 항목과 시기만
확장하므로 변경 범위가 통제되고, 앱이 제공하던 관리 기능을 잃지 않는다.

## 시기(PHASES) 구조 변경

### 현재
6단계, 전부 "예식 N개월 전"(`m`, 개월 수)로 정의.
일정 탭은 `pd.setMonth(pd.getMonth() - phase.m)`로 추천 시기를 계산한다.

### 변경 후
사진과 동일한 8단계로 재구성한다.

| id | 라벨 | 계산 기준 |
|----|------|-----------|
| p1 | 상견례 | ~12개월 전 (month) |
| p2 | 결혼 준비 시작 | 8~10개월 전 (month) |
| p3 | 웨딩데이 6~5개월 전 | 6개월 전 (month) |
| p4 | 웨딩데이 4~3개월 전 | 4개월 전 (month) |
| p5 | 웨딩데이 2~1개월 전 | 2개월 전 (month) |
| p6 | 웨딩데이 2주 전 | 14일 전 (day) |
| p7 | 웨딩데이 1주 전 | 7일 전 (day) |
| p8 | D-1 | 1일 전 (day) |

### 기술적 제약과 해결

막판 단계(2주/1주/D-1)는 "개월"로 표현되지 않는다. 따라서 시기 모델을
**개월 단위에서 개월+일 단위로 확장**한다.

- 각 phase는 `m`(개월) **또는** `d`(일) 중 하나의 오프셋을 가진다.
- `TimelineView`의 날짜 계산을 다음과 같이 분기한다:
  - `phase.d`가 있으면 `pd.setDate(pd.getDate() - phase.d)`
  - 없으면 기존대로 `pd.setMonth(pd.getMonth() - phase.m)`
- 라벨 포맷도 일 단위 단계에서는 "N월"보다 "D-N" 형태가 자연스러운지 확인한다.

## 카테고리 변경

기존 9개에 2개를 신설하여 **11개**로 확장한다.

기존: 예식장·플래너 · 스드메 · 본식 촬영·영상 · 예물·부케 · 한복·혼주 ·
청첩장 · 식순·인력 · 가족 행사 · 신혼여행

신설:
- **피부·뷰티** — 피부관리 시작, 다운타임 시술 금지 등
- **신혼 살림** — 신혼 가전 계약 등

흡수 분류한 항목:
- 하객 수 점검 · 사례비 봉투 · 본식 리허설 확인 → 식순·인력
- 웨딩 카페 가입 · 박람회/플래너 상담 → 예식장·플래너
- 청첩장 모임 → 청첩장

## 단계 분할 원칙

사진을 따라 일부 항목을 단계별로 분리한다 (앱이 항목별 업체·예산을 관리하므로
단계 분리가 자연스럽다):
- 드레스: 투어&계약(p2) → 가봉(p3) → 본식 피팅(p5)
- 신랑 예복: 투어&계약(p2) → 가봉(p3) → 픽업(p8)
- 혼주 한복: 계약(p3) → 픽업(p8)
- 청첩장: 제작(p4) → 발송(p5)

## 영향 받는 파일

- `src/constants.js` — `CATS`(2개 추가), `PHASES`(8단계 재정의 + `d` 필드),
  `seedItems`(약 46개로 재작성)
- `src/App.jsx` — `TimelineView`의 날짜 계산 분기(`m`/`d`), 라벨 포맷 확인

## 기존 룸 데이터 마이그레이션 (검토 필요)

Firestore에 이미 저장된 룸은 옛 항목·옛 phase id(p1~p6)를 그대로 보관한다.
`seedItems`/`PHASES`를 바꿔도 기존 룸의 저장된 `items`는 변하지 않는다
(신규 룸에만 새 seed 적용).

주의점: p1~p6의 **의미가 바뀌므로**(예: 기존 p6 "한 달 전" → 신규 p6 "2주 전"),
기존 룸의 항목이 잘못된 단계 라벨로 표시될 수 있다. 이 앱은 사실상 초기
단계이고 실데이터가 거의 없으므로, 마이그레이션 없이 진행하되 이 한계를
구현 계획에서 명시한다.

## 테스트 관점

- 일정 탭: 예식일을 설정했을 때 p1~p5(개월)와 p6~p8(일) 모두 올바른
  추천 시기가 계산되는가
- 체크리스트 탭: 11개 카테고리가 모두 정상 그룹핑되는가
- 예산 탭: 신규 카테고리 2개가 분류별 지출에 정상 합산되는가
- 신규 룸 생성 시 약 46개 항목이 시드되는가
